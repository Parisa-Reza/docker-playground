Without -d → Terminal shows logs (stuck)
With -d    → Terminal shows ID (free)

# Run container and expose a port
docker run -p 8000:8000 myapp   # Port 8000(Your Laptop)      --->   Port 8000 (Docker Container)
Port binding in Docker maps a port from your machine to a port inside the container so you can access container apps using localhost.
Same host port cannot bind to different conatiner port.

# VM vs Docker — Complete Guide

---

## What is a Virtual Machine (VM)?

A **Virtual Machine** is a software-based computer that runs inside your real computer.
It has its own full operating system, its own memory, its own storage — completely isolated.

Think of it like renting an entire apartment: you get your own kitchen, bathroom, bedroom — everything separate.

---

## What is Docker (Container)?

**Docker** packages just your app and the libraries it needs, then runs it on the host machine's OS kernel.

Think of it like renting a single room in a shared house: you share the kitchen and bathroom, but your room is yours.

---

## Side-by-Side Comparison

| Feature            | Virtual Machine (VM)       | Docker (Container)        |
|--------------------|---------------------------|---------------------------|
| Has its own OS?    | ✅ Yes (full OS)           | ❌ No (shares host OS)    |
| Startup time       | Minutes                   | Seconds                   |
| Size               | Gigabytes (10–50 GB)      | Megabytes (50–500 MB)     |
| Isolation          | Very strong               | Good (process-level)      |
| Performance        | Slower (overhead)         | Faster (near-native)      |
| Portability        | Heavy to move             | Lightweight, easy to ship |
| Best for           | Full OS isolation, legacy apps | Microservices, CI/CD, dev environments |

---

## The Hypervisor Layer

The **hypervisor** is the magic layer that makes VMs possible.
It sits between your physical hardware and the VMs, tricking each VM into thinking it has its own dedicated hardware.

### Type 1 — Bare Metal Hypervisor

```
[ Physical Hardware ]
        |
  [ Hypervisor ]        ← runs DIRECTLY on hardware (no host OS)
     /       \
[ VM 1 ]   [ VM 2 ]
```

- Runs **directly on the hardware**, no host OS underneath
- Faster and more efficient
- Used in **data centers and enterprise servers**
- Examples: **VMware ESXi**, **Microsoft Hyper-V**, **Xen**, **KVM**

---

### Type 2 — Hosted Hypervisor

```
[ Physical Hardware ]
        |
   [ Host OS ]          ← your normal OS (Windows/macOS/Linux)
        |
  [ Hypervisor ]        ← runs as an app on top of host OS
     /       \
[ VM 1 ]   [ VM 2 ]
```

- Runs **on top of a host OS** (like any other application)
- Easier to install and use
- Used on **personal computers and developer machines**
- Examples: **VirtualBox**, **VMware Workstation**, **Parallels**

---

## Full Architecture Stack

### VM Stack

```
+--------------------+   +--------------------+
|       App A        |   |       App B        |
+--------------------+   +--------------------+
|    Guest OS 1      |   |    Guest OS 2      |   ← Full OS per VM (~GBs)
| (Linux / Windows)  |   | (Linux / Windows)  |
+--------------------+   +--------------------+
|              Hypervisor                     |   ← Tricks VMs into thinking
|         (Type 1 or Type 2)                  |     they own the hardware
+---------------------------------------------+
|           Physical Hardware                 |
+---------------------------------------------+
```

### Docker Stack

```
+------------------+   +------------------+
|   Container A    |   |   Container B    |
|  (App + libs)    |   |  (App + libs)    |   ← No OS, just app + dependencies
+------------------+   +------------------+
|           Docker Engine                  |   ← Manages containers
+------------------------------------------+
|        Host OS (Shared Kernel)           |   ← One OS for everyone
+------------------------------------------+
|          Physical Hardware               |
+------------------------------------------+
```

---

## Key Takeaway

| Concept      | Simple Explanation                                              |
|--------------|-----------------------------------------------------------------|
| **VM**       | A full computer inside your computer. Heavy but very isolated.  |
| **Docker**   | A lightweight box for your app. Shares the OS, fast and small.  |
| **Hypervisor** | The layer that tricks VMs into thinking they own the hardware. |
| **Type 1**   | Hypervisor on bare metal → used in servers/cloud.               |
| **Type 2**   | Hypervisor on top of your OS → used on laptops/desktops.        |

---

## When to Use Which?

- Use **VMs** when you need full OS isolation, run different OS types, or have legacy software.
- Use **Docker** when you want fast, lightweight, portable app deployment (microservices, CI/CD pipelines, dev environments).

1. What is a Docker Volume?

A Docker volume is a persistent storage location managed by Docker.

👉 It stores data outside the container lifecycle.

Container deleted ❌ → Data lost (no volume)
Container deleted ❌ → Data safe (with volume) ✅
2. Who manages volumes?

👉 Docker Engine manages volumes, not your app, not the container.

Docker creates it
Docker stores it on host machine
Docker attaches it to containers
App → writes data
Container → passes data
Docker → stores it in volume
3. Types of Docker Volumes
(1) Named Volume (MOST IMPORTANT)
pgdata:/var/lib/postgresql/data
Features:
Managed by Docker
Best for databases (Postgres, MySQL)
Survives container deletion
(2) Anonymous Volume
docker run -v /data postgres
Features:
No name
Docker creates random ID
Hard to manage
Managed by Docker
Mostly temporary use
(3) Bind Mount (Host Path)
./data:/var/lib/postgresql/data
Features:
Uses your laptop folder
Good for development
Not ideal for production DB
Managed by host os
4. Volume mapping (important concept)
HOST STORAGE  →  CONTAINER PATH

Example:

pgdata:/var/lib/postgresql/data

Means:

Docker volume (pgdata)
        ↓
Postgres data folder inside container

version: "3.9"

services:
  web:
    build: .
    ports:
      - "8000:8000"

  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:


here, pgdata:/var/lib/postgresql/data means Docker stores Postgres database files from the container’s data directory into a persistent Docker-managed volume called pgdata.


Docker Volume Types (List)
Named Volume
Managed by Docker
Example: mydata:/app/data
Bind Mount
Managed by Host OS (your laptop)
Example: ./data:/app/data
Anonymous Volume
Auto-created by Docker with random name
Example: /app/data (no name given)


why there are 2 volumes in the code ??

1. Inside the service (usage)
volumes:
  - pgdata:/var/lib/postgresql/data

👉 This is mounting a volume to the container

Meaning:

pgdata (Docker storage)
   ↓
/var/lib/postgresql/data (inside Postgres container)

This is where Postgres stores its database files.

2. At the bottom (definition)
volumes:
  pgdata:

👉 This is declaring the volume

Meaning:

“Hey Docker, create and manage a volume named pgdata”
3. Why both are needed

Think like this:

Step 1: Declare storage
Create a disk called pgdata
Step 2: Attach storage
Connect pgdata → Postgres data folder

The first “volume” attaches storage to Postgres, and the second “volume” declares and creates that storage in Docker.

1. Bridge network (DEFAULT)
What it is:

Containers talk to each other through a virtual private network inside Docker.

Container A  ←→  Container B
        (bridge network)
Example:
docker run postgres

👉 Automatically uses bridge network

Key idea:
Containers get private IPs
Can communicate using container names
Isolated from host
Real meaning:
Docker creates a virtual switch (bridge)
Location (internally):
/var/lib/docker/network/
docker0 bridge interface
2. Host network
What it is:

Container uses your laptop’s network directly

No isolation between host and container
Example:
docker run --network host nginx
Meaning:
Container port = Host port
localhost:80 → container:80
Key idea:
No port mapping needed (-p ignored)
Fast performance
No network isolation
Real meaning:
Container is directly attached to your machine network
3. None network (NULL)
What it is:

Container has NO network at all

No internet
No localhost
No communication
Example:
docker run --network none alpine
Key idea:
Fully isolated container
No inbound/outbound traffic
Only file/process execution

* Dockerizing an app

FROM → choose base image (start system)
WORKDIR → where you work inside container
COPY    → bring files in
RUN     → install/setup commands
EXPOSE  → show port used by app
ENV     → set environment variables
CMD     → start app

=========================
FROM python:3.12

# Set working directory inside container
WORKDIR /app

# Copy project files into container
COPY . .

# Set environment variables
ENV PYTHONUNBUFFERED=1
ENV APP_ENV=development

# Install dependencies
RUN pip install -r requirements.txt

# Expose port (informational, not publishing)
EXPOSE 8000

# Run application
CMD ["python", "app.py"]
